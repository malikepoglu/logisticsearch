## EN 9. Quality is not only a line-count threshold

The project quality standard is not a line-count trick. A document can satisfy a minimum line threshold and still fail if it does not explain the system, preserve context, or teach the reader how to continue safely.

For README, canonical, and runbook documents, the expected quality gate includes all of the following:

- coverage of the real project scope;
- coverage of current operational truth;
- coverage of historical context when history affects current decisions;
- coverage of safety boundaries;
- coverage of forbidden mutations;
- coverage of the next safe action;
- enough beginner-level explanation that a new reader can understand the system without private chat memory;
- enough operator-level precision that a command, path, or runtime boundary cannot be misunderstood;
- enough GitHub continuity that future work can resume from the repository alone.

The `preferably preferably 600+ nonblank English lines when the document scope justifies that depth when the document scope justifies that depth` and `preferably preferably 600+ nonblank Turkish lines when the document scope justifies that depth when the document scope justifies that depth` rule is only a floor for rich hub documents. It is not a replacement for semantic quality. A low-quality repetitive file must fail even if it is long. A high-quality conversion must preserve or improve the old document's explanation depth.

Canonical docs and runbooks follow the same principle. If a canonical document defines truth, it must explain why that truth exists. If a runbook can mutate a host, service, runtime, network, or hardware state, it must explain the safety gate before the command. No DB mutation. No pi51c mutation. No live runtime mutation. No systemd mutation. No crawler start. No control script execution unless a later explicitly approved mutating runbook step says otherwise.


## EN 10. Book-style readability rule

Book-style numbering is the preferred readability model.

The `600+` line target is a preferred depth target for large README, hub, canonical architecture, and broad runbook documents. It is not a blind universal minimum for every Markdown file. Smaller focused subjects, such as a narrow `playwc`, `pausewc`, or single-control explanation, may be shorter when the subject is genuinely small. The non-negotiable rule is not raw length; the non-negotiable rule is complete scope coverage, beginner-first clarity, operator safety, exact current truth, exact historical truth, and enough detail for a future maintainer to continue from GitHub alone without private chat memory.


Numbering must carry the structure directly.

Avoid artificial labels that pretend to add depth but do not add meaning.
Prefer direct numbered book-style lines because the number already shows the position, while the sentence carries the actual meaning.

Avoid repeated bold prefixes that imitate structure without becoming real structure.

Use direct numbered headings when a point deserves its own structural place.

Correct English format:

    EN 12. Immediate next action
    EN 12.1 Preserve beginner-first explanations around every major contract.
    EN 12.2 Preserve operator-first safety language around every mutating action.
    EN 12.3 Keep GitHub sufficient for continuity without private chat memory.

The same rule applies to Turkish with the same numbering and the same structural order.

A document is detailed because the explanation is useful, scoped, beginner-friendly, and operationally clear.

It is not detailed because it repeats decorative labels.

The target reading feel is a high-quality practical how-to book: numbered headings, meaningful subheadings, short paragraphs, clear constraints, and visible safety boundaries.

README, canonical ve runbook dokümanları için beklenen kalite gate şu başlıkların tamamını içerir:

- gerçek proje kapsamının kapsanması;
- current operational truth bilgisinin kapsanması;
- geçmiş kararlar bugünkü kararı etkiliyorsa historical context bilgisinin kapsanması;
- safety boundaries bilgisinin kapsanması;
- forbidden mutations bilgisinin kapsanması;
- next safe action bilgisinin kapsanması;
- private chat memory olmadan yeni okuyucunun sistemi anlayabileceği beginner-level açıklama;
- command, path veya runtime boundary yanlış anlaşılmayacak kadar operator-level kesinlik;
- gelecekteki işin yalnızca repository üzerinden devam edebilmesini sağlayan GitHub continuity.

`preferably 600+ nonblank English lines when the document scope justifies that depth` ve `preferably 600+ nonblank Turkish lines when the document scope justifies that depth` kuralı yalnızca rich hub dokümanları için taban eşiğidir. Semantic quality yerine geçmez. Düşük kaliteli ve tekrarlı bir dosya uzun olsa bile fail etmelidir. Kaliteli conversion eski dokümanın açıklama derinliğini korumalı veya iyileştirmelidir.

Canonical docs ve runbooklar da aynı ilkeye uyar. Bir canonical document truth tanımlıyorsa o truth’un neden var olduğunu açıklamalıdır. Bir runbook host, service, runtime, network veya hardware state değiştirebiliyorsa command öncesinde safety gate’i açıklamalıdır. No DB mutation. No pi51c mutation. No live runtime mutation. No systemd mutation. No crawler start. No control script execution; bunlar yalnızca ileride açıkça onaylanmış mutating runbook adımı varsa değişebilir.
